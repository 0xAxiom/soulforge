# CrewAI

**Target:** https://docs.crewai.com — https://github.com/crewAIInc/crewAI

## What it is

CrewAI is a Python multi-agent orchestration framework built on a single metaphor: agents are team members with roles. You define agents (Researcher, Writer, Reviewer) with role, goal, and backstory, assemble them into Crews that execute tasks collaboratively, and wrap Crews in Flows that add deterministic control logic on top. As of May 2026 it claims 12M daily executions and 60% Fortune 500 adoption.

## Architecture

- **Agent triad — role / goal / backstory.** Every agent is defined by three natural-language fields that shape its system prompt. Role is function, goal is objective, backstory is personality and context. This is structured prompt engineering; the triad is injected at call time, not stored separately.
- **Crew = collaborative task pool.** A Crew is a set of agents plus a set of tasks. Tasks route to agents by match. Two execution modes: sequential (task chain) and hierarchical (manager agent dispatches to workers). The LLM drives task delegation in hierarchical mode.
- **Flow = orchestration wrapper.** Flows sit above Crews and provide deterministic control using three Python decorators: `@start`, `@listen`, and `@router`. Flow state is a Pydantic model. Execution is event-driven: `@listen(method_name)` fires when that method completes. `@human_feedback` pauses for approval. This is the layer that makes CrewAI safe for production.
- **Unified memory with composite scoring.** The v2 memory API replaced separate short-term/long-term/entity memory with one `Memory` class. Retrieval ranks candidates using three weighted factors: semantic similarity (embedding distance), recency (exponential decay), and importance (LLM-assigned at write time). Backend defaults to LanceDB; any `StorageBackend` protocol implementation works.
- **Background saves.** Memory writes are async and non-blocking. Agents don't wait on persistence; the save queue drains independently. This eliminates memory latency from the agent's hot path.
- **MCP and A2A native (v1.10+).** CrewAI agents can expose themselves as MCP servers or participate in A2A (Agent-to-Agent) handoffs with a single config flag. No custom protocol work required.

## What soulforge can learn

- **Composite recall scoring belongs in memory/recall.** Soulforge's `SqliteRecallStore` uses deterministic local hash embeddings — effectively a content hash, not a semantic embedding. When a real embedding backend is wired in, scoring should weight recency and importance alongside similarity. The pattern: `score = w_sem * sim + w_rec * recency_decay(age) + w_imp * importance`. See `memory/src/recall.ts` — the ranking function is currently a stub.
- **Make the orchestrator/worker split a named pattern.** CrewAI's Flow/Crew split (deterministic orchestration vs. LLM autonomy) is the same thing soulforge calls the "control layer vs. soul layer," but CrewAI named it explicitly and built tooling around it. The `.ai/patterns.md` and `docs/ARCHITECTURE.md` should document this as a first-class soulforge pattern: when a task is deterministic, it should live in an endpoint or flow; when it requires judgment, it routes to a soul.
- **Role + Goal + Backstory = Identity + Voice + Values.** The CrewAI triad is a compressed version of what soulforge souls express in full sections. The mapping matters because it confirms the soulforge soul schema is structurally correct — just richer. The advantage: soulforge souls are editable by non-engineers; CrewAI agent definitions are Python classes.
- **Worker souls need explicit input/output contracts.** CrewAI agents in a hierarchical crew receive structured task objects from the manager. Soulforge's `# Signature` section covers this, but there are no example souls showing a worker that gets dispatched by an orchestrator. The gap: specialist souls that declare "I accept task X, I return Y" and make no assumptions about what called them.

## What soulforge should NOT copy

- **Backstory as personality shortcut.** The backstory field is a natural-language string injected into the system prompt. It works, but it's opaque — there's no way to audit, version, or test what the backstory actually does to model behavior. Soulforge's Voice and Values sections force that reasoning into legible, testable form.
- **The crew metaphor for everything.** Flat teams (all agents are peers) work for simple report-writing pipelines. They break down when you need hierarchical authority, agent trust levels, or scoped economic permissions. Soulforge should stay with explicit soul-level scope and refuses rather than flattening those distinctions into role strings.
- **LanceDB as default.** LanceDB is a good vector store for Python but introduces a native dependency. For a TypeScript substrate repo like soulforge, SQLite + deterministic hash embeddings is more portable, auditable, and CI-friendly. When real semantic recall is needed, the backend should be swappable (which soulforge's `StorageBackend` protocol already supports).
- **Decorator-heavy orchestration in application code.** `@start`, `@listen`, `@router` are ergonomic in a monolithic Python app but don't map cleanly to TypeScript endpoints or to agents coded by an LLM. Soulforge's "endpoint as the flow" pattern keeps orchestration in a typed function, not in decorator metadata.
