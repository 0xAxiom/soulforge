## What it is

DSPy (Declarative Self-improving Python) is Stanford's framework for *programming* language models rather than prompting them. Instead of writing prompt strings, you declare typed input/output contracts (Signatures), compose them into Modules, and run an Optimizer that automatically tunes your prompts and/or model weights against a metric function you define. The compile-then-run model separates expensive optimization from cheap inference, analogous to a compiled language — you pay the optimization cost once, then deploy a frozen, optimized program.

---

## Architecture

**Three core abstractions:**

- **Signature** — a declarative input/output contract, defined as a Python class with typed `InputField` / `OutputField`. Separates "what the task is" from "how the model is instructed." This is the unit that optimizers rewrite; your program logic stays stable.
- **Module** — encapsulates a prompting strategy (ChainOfThought, ReAct, ProgramOfThought) and is signature-agnostic. You can swap strategies without touching program logic. Modules are composable: a module's output can feed another module's input, building pipelines as call graphs.
- **Optimizer (formerly Teleprompter)** — takes (program, trainset, metric) and tunes program parameters (prompts + weights) to maximize the metric. Several strategies exist:
  - `BootstrapFewShot` — runs the program on training inputs, collects behavioral traces, filters traces that pass the metric, and embeds them as few-shot demonstrations. Effective with ~10 examples.
  - `MIPROv2` — a three-stage optimizer: bootstrap traces → propose instruction candidates (grounded in code + data + traces) → Bayesian search over (instruction × few-shot) combinations. Best at 200+ examples.
  - `BootstrapFinetune` — distills prompt-based programs into fine-tuned weights. Pre-inference cost traded for zero inference overhead.
  - Optimizers are composable: chain them, ensemble top-N outputs.

**Compile-then-run paradigm:**
The compilation step (optimizer run) is expensive and done offline. The output is a *frozen program* with optimized prompts, saved and loaded for inference. This means "prompt engineering" becomes an automated, offline, reproducible step, not a live debugging session.

**Metrics are first-class modules:**
DSPy metrics are plain Python functions (or DSPy modules themselves). `dspy.Evaluate` wraps them with parallelism and cost control. Built-ins include `SemanticF1` (long-form), `answer_exact_match`, `answer_passage_match`. Custom metrics are lambdas. A metric that uses another LM for judgment (LLM-as-judge) is just another DSPy module — the eval system is recursive and composable.

**Trace collection is built into the runtime:**
When you run your program, DSPy records every module invocation with its inputs, outputs, and whether the metric passed. This trace is the raw material for BootstrapFewShot — the optimizer does not need hand-labeled data for demonstrations, it bootstraps them from successful runs.

## What soulforge can learn

**1. Metrics belong in the eval definition, not in prose documentation.**
Soulforge's eval README lists "open design questions" about scoring rubrics. DSPy's answer is concrete: write a metric function that takes `(example, prediction) -> float|bool`. That function *is* the design decision. The prose is secondary. Soulforge v2 eval should define each golden test's acceptance criterion as a typed assertion, not a human-readable description.
→ Applies to: `eval/goldens/` format, `eval/score/` module

**2. Trace-to-demonstration loop closes the soul-eval cycle.**
Soulforge's architecture diagram shows soul → tools → eval as a one-way chain. DSPy shows it should be a loop: trace every agent turn → filter turns where output met the metric → use passing traces as few-shot demonstrations in the next soul revision. This is soul improvement through behavioral evidence, not intuition.
→ Applies to: `eval/traces/` format, soul revision workflow

**3. Signature = typed soul scope.**
Soulforge souls declare `scope` and `refuses` in markdown prose. DSPy Signatures show that input/output types are a better primitive: they're machine-readable, optimizable, and testable. Soulforge souls could gain a `signature:` frontmatter block with typed `inputs` and `outputs` — not to make souls code, but to make the soul's contract explicit enough that eval can auto-generate goldens from it.
→ Applies to: `souls/schema/soul.schema.json` (future schema bump)

**4. Cost-in-the-loop is a solved problem in DSPy.**
DSPy's `dspy.Evaluate` parallelizes golden runs with configurable concurrency and caches intermediate results. Soulforge's eval README calls this an "open design question." DSPy's answer: run goldens in parallel (not sequentially), cache trace results with a content-addressed key (hash of input + soul version), and cap nightly CI runs rather than per-PR. This prevents the $100/run problem.
→ Applies to: `eval/` harness design, CI integration

**5. LLM-as-judge plus deterministic assertions — not either/or.**
DSPy uses both in practice: exact match for factual fields, SemanticF1 for prose, and LLM judge for open-ended quality. Soulforge eval should adopt the same tiered approach: hard assertions for refusal conditions (deterministic), LLM judge for voice/tone adherence (fuzzy), exact checks for structured outputs.
→ Applies to: `eval/score/` scoring module design

## What soulforge should NOT copy

**The Python-specific compilation model.** DSPy's power is inseparable from Python's introspection: it traces module calls, inspects signatures, rewrites prompts via in-process optimizer loops. Soulforge is TypeScript + markdown. An attempt to port DSPy's optimizer would be a framework, which CLAUDE.md explicitly forbids. The *ideas* (metrics-first, trace-to-demo, compile-then-run) are portable; the implementation is not.

**Teleprompter complexity for v1.** DSPy's MIPROv2 with Bayesian search is designed for teams with 200+ labeled examples running systematic ML pipelines. Soulforge's v2 eval starts with hand-curated goldens. Don't skip to MIPROv2-style automated instruction search before there are enough golden cases to make optimization meaningful.

**Tight coupling between metric and trainset.** DSPy works best when training data and metric share the same domain. Soulforge's souls are general-purpose — a refusal checker for one soul is meaningless for another. Metrics must be soul-specific, not global. Don't build a single eval harness that tries to score all souls the same way.

**Optimizing the soul vs. understanding it.** DSPy optimizes prompts automatically. Soulforge's bet is that souls should be human-readable and human-editable. Automated soul rewriting undermines the promise that "a soul can be edited by a non-engineer." Use DSPy's tracing insights to *inform* human soul edits, not to replace them.
