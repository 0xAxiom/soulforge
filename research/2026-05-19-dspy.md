# DSPy — Declarative Self-Improving LM Programs

**Target:** Stanford NLP group's DSPy framework (v2.5+)
- Repo: https://github.com/stanfordnlp/dspy
- Docs: https://dspy.ai
- Paper: "DSPy: Compiling Declarative Language Model Calls into Self-Improving Pipelines" (Khattab et al., 2023)

## What it is

DSPy is a framework for building LLM programs as composable, optimizable pipelines. Instead of writing prompts as strings, you write code that declares *what* each step receives and produces (a Signature), applies a *how* (a Module like ChainOfThought or ReAct), and then runs an optimizer offline to tune the prompts and few-shot examples against a metric function. The resulting artifact is a compiled, optimized program — not a hand-crafted prompt.

The core bet: "Programs are code, not strings." The optimizer finds the instructions; you define the architecture and the metric.

## Architecture

- **Signatures** — typed I/O contracts for a single LM call. Can be terse (`"context, question -> answer: bool"`) or class-based with `InputField`/`OutputField` and type hints. Type hints enforce output shape (list, float, bool, custom type) across modules. The signature is the composition surface: any module that accepts the same signature is a drop-in replacement.
- **Modules** — reusable prompting technique + signature. `Predict` (direct), `ChainOfThought` (step-by-step reasoning), `ProgramOfThought` (code-generating), `ReAct` (tool-calling), `RLM` (sandboxed REPL). Modules are stateless; they store optimized instructions and demos after compilation, but carry no session state.
- **Optimizers (teleprompters)** — offline algorithms that adjust a program's prompts and/or weights to maximize a user-supplied metric. Key ones: `BootstrapFewShot` (synthesizes effective few-shot demos automatically), `MIPROv2` (multi-stage instruction search, current flagship), `GEPA` (reflective prompt evolution using LLM-as-optimizer), `BootstrapFinetune` (distills prompt logic into model weights).
- **Metric function** — a Python function scoring any `(input, output)` pair. Not a fixed test format. The optimizer runs the program against training examples, scores outputs, and adjusts accordingly. The metric *is* the success criterion — it must be explicit before you optimize.
- **Compile-then-run** — optimization is expensive and offline. The compiled program (with its learned instructions + demos) is a deployable artifact. Inference is cheap and does not re-optimize. This separation lets you iterate on architecture separately from prompt quality.
- **Global configuration** — models initialized via `dspy.LM`, configured globally via `dspy.configure()`. Simplicity over explicit dependency injection.
- **Training split convention** — 20% train / 80% dev. Reversed from ML convention because: overfitting on tiny training sets is the bigger risk; you need dev set to catch optimizer overfit.

## What soulforge can learn

- **Signature section in souls.** DSPy's typed I/O contracts are the strongest architectural idea here. Soulforge souls have `scope` (prose intent) but no machine-readable input/output contract. Adding an optional `# Signature` section (markdown table of typed inputs and outputs) makes souls composable artifacts — coding agents know exactly what a soul accepts and produces, enabling wiring chains of souls without guessing at the interface. This is especially valuable for deterministic-workflow souls where every step's handoff record should match.
- **Metric as a first-class soul citizen.** DSPy makes the success metric explicit before any optimization happens. Soulforge's eval goldens already contain `criteria` — this reinforces that design. But the soul itself doesn't reference its success criteria. A soul that names its metric (even as a prose statement in `# Limits` or a new `# Criteria` section) would be more eval-friendly: the eval harness can surface criterion mismatches at soul-review time, not after running goldens.
- **Compile vs. run as a naming distinction.** Soulforge's eval/ loop (run goldens → score → diff soul versions) is *exactly* DSPy's compilation loop, just without automatic prompt rewriting. Naming this explicitly in ARCHITECTURE.md as the "eval-as-optimizer" pattern would help coding agents understand what eval/ is for: it's not just QA, it's the mechanism for deciding which soul version ships.
- **Stateless modules as a design constraint.** DSPy's modules are explicitly stateless; conversation history requires deliberate `dspy.History` threading. Soulforge already separates memory/ from souls/ — this is the same principle. Worth reinforcing in docs: a soul is a stateless policy; memory/ is the only place state lives.
- **20/80 train/dev split intuition.** For soulforge eval goldens, this translates to: keep most goldens as regression guards (dev), use a small set as "training examples" when running an LLM judge in optimization mode. Don't burn all goldens as training data.

## What soulforge should NOT copy

- **Global `dspy.configure()`.** Convenience for prototyping, architectural smell at scale. Soulforge's explicit module-level wiring (soul → tool → endpoint → memory) is cleaner than invisible global model injection. Don't add a global `soulforge.configure()`.
- **Automatic prompt rewriting.** DSPy's optimizers change the prompts without human review. For agent policy (souls), this is unsafe — you can optimize the soul into something that's technically high-scoring but behaviorally wrong in ways the metric doesn't capture. Soulforge's current model (human authors the soul, eval validates it, diff surfaces regressions) is safer for policy work. Automated rewriting belongs in application layers, not soul templates.
- **String-shorthand signatures.** The terse `"context -> answer"` string form is ergonomic for simple cases but opaque to tooling. Soulforge should stick to explicit markdown tables for the typed I/O pattern — human-readable first.
- **Module hierarchy as the composition unit.** DSPy builds composition via Python class inheritance (`class MyProgram(dspy.Module)`). Soulforge uses files and markdown, not runtime objects. Resist the temptation to add a Soul class or runtime module hierarchy — the repo structure *is* the composition surface.
