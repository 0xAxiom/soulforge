# Letta (formerly MemGPT) — Stateful Agents with Tiered Memory

**Target:** https://github.com/letta-ai/letta · https://docs.letta.com · https://www.letta.com/blog/letta-v1-agent

## What it is

Letta (rebranded from MemGPT) is an agent runtime where memory is a first-class primitive — persistent, tiered, and self-editable by the agent. The core thesis is that LLMs are stateless, so the runtime must manage state explicitly, borrowing the OS metaphor of RAM (in-context), disk cache (recall), and cold storage (archival). It runs as a server you deploy and hit via REST API or SDK.

## Architecture

- **Three-tier memory hierarchy.** Core memory = in-context labeled blocks (character-capped, always present). Recall memory = searchable conversation history stored on disk (grows unbounded, retrieved on demand). Archival memory = external vector/graph databases (semantic search, cold retrieval). The tiers differ on the immediacy-vs-capacity axis, not on semantic importance.
- **Labeled memory blocks.** Core memory is not a blob — it's a set of named containers: `label`, `description`, `value`, `char_limit`. Each block has a declared purpose. The agent edits blocks via tool calls, not by rewriting the system prompt.
- **Self-editing memory via tool calls.** Every memory mutation is an explicit tool invocation. This makes memory writes observable, replayable, and eval-able. No hidden state drift.
- **Agent-loop evolution: heartbeat → native reasoning.** Original MemGPT used a `request_heartbeat` tool parameter to let the agent signal "I need another turn." Letta v1 dropped this in favor of model-native reasoning tokens (Claude's thinking, GPT's o-series reasoning). Gain: better performance on frontier models. Loss: you can't observe or control reasoning on closed APIs.
- **MemFS — git-tracked memory.** Latest addition: memory blocks can be git-tracked files, giving you versioned memory history. Interesting for auditability; currently local only.
- **Platform-first, not framework-first.** Letta runs a stateful agent server. Agents live in the server, not in your process. This is architecturally the opposite of soulforge's substrate approach.

## What soulforge can learn

- **Declare memory blocks explicitly in souls.** The `# Memory` section in starter-soul is vague ("None in v1"). Letta shows that naming blocks, their purpose, and their capacity constraint makes memory legible — to humans and to coding agents writing the soul. A soul should say: `user-context (char_limit: 2000) — what I know about this user` not just "short-term memory."
- **Self-editing memory = tool-call pattern.** When an agent needs to update its own durable state, that update should be a typed tool call with a receipt, not a side effect buried in a tool's implementation. Soulforge's economic tool boundary rule applies to memory too: `soul policy → memory-write tool → long-term store → receipt → observability`.
- **Tiered promotion should be triggered by explicit conditions.** Letta's three tiers map well to soulforge's short-term / recall / long-term split. The gap in soulforge's current design: there's no documented condition for *when* to promote. Letta makes promotion an agent action (tool call to `archival_memory_insert`). Soulforge should add trigger conditions to `memory/README.md`.
- **Character limits as a design constraint, not a failure.** Forcing agents to summarize rather than append is valuable. Soulforge could add an optional `char_limit` field to long-term memory blocks as a forcing function for compression.

## What soulforge should NOT copy

- **The stateful agent server model.** Letta runs a persistent server process that holds all agent state. This buys cross-session persistence but costs operational complexity. Soulforge's local-first SQLite approach is simpler and more auditable for the cases soulforge targets. The server model is the right choice when you need multi-tenant, cross-session agents at scale; it's wrong for single-developer substrate exploration.
- **Opaque native reasoning.** Letta v1 embraces model-native reasoning tokens for performance. The tradeoff is you lose the ability to inspect or replay the reasoning trace. Soulforge's eval primitive depends on observable traces — opaque reasoning breaks eval replay. Avoid hiding reasoning in provider-controlled tokens when soulforge's use case requires trace auditability.
- **The OS metaphor as user-facing language.** Letta's "RAM/disk/cold storage" framing is clarifying internally but alienating as user-facing copy for agent developers who don't want to think about operating systems. Soulforge's short-term / recall / long-term naming is already more intuitive.
