# Magentic — LLM-as-Typed-Function

**Target:** https://magentic.dev — https://github.com/jackmpcollins/magentic — https://pypi.org/project/magentic/

## What it is

A Python library (not a framework) that turns LLM calls into regular typed functions via decorators (`@prompt`, `@chatprompt`, `@prompt_chain`). The caller defines input parameters and a return type annotation; Magentic handles the API call, response parsing, and Pydantic validation. No agent runtime, no graph, no workflow engine — just functions that happen to be backed by a model.

## Architecture

- **Decorator-as-contract**: `@prompt("Do X with {arg}")` + `def fn(arg: str) -> MyModel: ...` — the return type IS the output contract, not a docstring, not a README. Pydantic validates automatically. Failed validation → auto-retry with the error text injected as feedback.
- **Three decorator levels**: `@prompt` (one-shot, typed output), `@chatprompt` (multi-turn with system + few-shot messages), `@prompt_chain` (automatic tool-call loop — LLM calls registered `functions` until it returns a non-FunctionCall).
- **Composability over orchestration**: LLM functions compose like regular Python functions. No coordinator. No graph. Pass an LLM function as a `functions=` argument to another; the inner call becomes a tool. Complexity scales with composition, not framework features.
- **Async-native**: `async def` functions get concurrent execution automatically. Multiple LLM calls in-flight simultaneously — 7× speedup cited for batch cases.
- **Multi-backend**: OpenAI, Anthropic, Ollama, LiteLLM, Mistral — swappable via config without touching function code.
- **OTel from day 1**: Native OpenTelemetry + Pydantic Logfire integration. Not an afterthought. Every decorated function emits trace spans.
- **No agent runtime**: There is no `Agent` class, no `AgentExecutor`, no session object. State is caller-managed. The library is pure function-wrapping.

## What soulforge can learn

- **Endpoints are typed functions.** Magentic's strongest insight: the input schema and return type ARE the contract. Soulforge endpoints should declare Zod input/output schemas at the top of every template, not buried in handler code. The schema is the spec; everything else is implementation. This directly tightens the `api-key-endpoint` template (see `endpoints/templates/api-key-endpoint.md`).
- **Retry-on-validation belongs in the primitive.** Magentic bakes validation-retry into every decorated call: failed Pydantic → inject error → retry up to budget. The `typed-output-soul.md` already documents this pattern. Endpoint templates should make it explicit for the `mode: ai` handler path too — not just soul policy, but implementation guidance.
- **Few-shot via `@chatprompt` as soul pattern.** Magentic's `@chatprompt(SystemMessage(...), UserMessage(...), AssistantMessage(...))` is structured few-shot injection. Soulforge souls do this as free-form markdown sections. Worth documenting the tradeoff: free-form is more readable, structured injection is more reliable for tight output constraints.
- **Async-native evaluation.** Soulforge eval currently treats cases serially. Magentic's concurrent async pattern applies directly: run all golden cases concurrently rather than sequentially. Real speedup for eval harnesses with > 10 cases.
- **OTel from day 1, not week 6.** Every endpoint template should wire observability before the handler, not as an optional add-on. Magentic demonstrates this is a library-level concern, not an application concern.

## What soulforge should NOT copy

- **Decorator magic.** `@prompt` is Python-specific and hides what's happening. In TypeScript (soulforge's primary language), decorator magic creates debugging friction. Soulforge's explicit Zod schemas + handler functions are better: visible, testable, type-safe without annotation gymnastics.
- **Implicit tool loop (`@prompt_chain`).** Hiding the tool-call loop is fine for demos; it's dangerous for production. No cost cap, no iteration limit (beyond `max_retries`), no observability at each step by default. LangGraph's explicit state machine is safer for agents that have economic consequences or can get stuck. Soulforge should keep tool loops explicit.
- **Pure-function framing for all LLM calls.** Magentic's model assumes side-effect-free functions. Soulforge endpoints have economic effects (payments, onchain writes). Pure function framing breaks the boundary. Soulforge's explicit `side_effects` field in endpoint manifests is correct; Magentic's approach would erase that discipline.
- **No multi-turn session management.** Magentic delegates session state to the caller entirely. That's fine for a library; soulforge's `memory/` primitive exists precisely to give agents durable session state. Don't treat memory as optional.
