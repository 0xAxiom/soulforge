# Agno (formerly Phidata)

**Target:** https://agno.com — https://github.com/agno-agi/agno — https://docs.agno.com/

## What it is

Agno (rebranded from Phidata in January 2025) is an open-source Python framework for building multi-modal agents with a strong production runtime story. Its tagline is "simple, fast, model-agnostic." The key stat: ~2μs agent instantiation, ~3.75 KiB memory footprint per agent — achieved by being fully stateless at the SDK layer. Teams ship with three coordination modes (route, coordinate, collaborate). The framework pairs with an optional "AgentOS" control plane that adds sessions, auth, RBAC, tracing, and horizontal scaling without changing agent code.

## Architecture

- **SDK ≠ runtime.** The SDK is a pure Python library for building agents. AgentOS is a separate optional production layer that wraps the SDK as a FastAPI service with 50+ endpoints, SSE/WebSocket streaming, and Redis-backed horizontal scaling. Developers can use the SDK without the runtime — this is the rare framework that makes the seam explicit.
- **Three team coordination modes.** Agno formalizes: (1) **route** — pick exactly one specialist based on the query, dispatch to it; (2) **coordinate** — orchestrator breaks work into parts, dispatches to specialists, assembles results; (3) **collaborate** — all team members see the task and contribute additively, no single coordinator. This is a cleaner naming than sequential/parallel/shared-state; the axis is "who controls dispatch" rather than "how does state flow."
- **Knowledge ≠ Memory.** Agno makes this distinction explicit in its primitives: **Knowledge** is a curated, vectorized corpus (PDFs, docs, sites) that the agent retrieves from via RAG — it doesn't change session-to-session. **Memory** is per-user/per-session accumulated facts from interactions. **Storage** is the raw CRUD persistence layer underneath both. These are three separate classes, not one unified recall store.
- **Stateless-by-design.** The performance numbers (2μs, 3.75 KiB) come from not building a runtime state machine. Each agent invocation is constructed fresh from config. Session continuity lives in the storage layer, not in agent objects. This is the mechanism behind horizontal scaling — there's nothing to shard.
- **DB abstraction via BaseDb.** All primitives (Agent, Team, Workflow) write through a shared `BaseDb`/`AsyncBaseDb` interface. Swapping from SQLite to PostgreSQL to DynamoDB is a one-line config change. The agent code never touches the storage driver.
- **Multi-modal native.** Text, image, audio, video, and document inputs are first-class. The framework doesn't treat multi-modality as a special case — it's in the core `Agent` class.
- **Observability built in.** OpenTelemetry export out of the box. Traces, spans, and structured events are automatic, not instrumented manually.
- **~40k GitHub stars, active.** Not a toy — large community, rapid iteration as of May 2026.

## What soulforge can learn

- **Name "knowledge retrieval" as distinct from memory.** Soulforge's `recall.ts` is populated by `reflect.ts` from session transcripts — it's accumulated interaction memory, not curated document retrieval. When a developer wants to build an agent that answers from a corpus (product docs, policies, reference material), soulforge has no guidance for this. It should: explicitly name "knowledge retrieval" as a pattern, distinguish it from session memory, and direct developers toward a retrieval tool call (not SqliteRecallStore). A `knowledge-grounded-soul.md` example would demonstrate the pattern. See concrete addition below.
- **"Route" as a named coordination mode.** Soulforge has `handoff-router-soul.md` (classify + route + stop) and `workflow-orchestrator-soul.md` (coordinate multiple specialists). What's missing is the lighter "route and pass through" mode where the dispatcher picks a specialist but the result flows back through it — as opposed to a terminal handoff. Naming this in `docs/ARCHITECTURE.md` clarifies when to reach for `handoff-router` vs a lighter routing-only wrapper.
- **Stateless default as a documented invariant.** Agno's performance comes from stateless construction. Soulforge already has no runtime, but it doesn't say *why*. Adding "agents have no startup state — all context comes from the call" as an explicit principle in `docs/ARCHITECTURE.md` would help coding agents avoid accidentally building stateful singletons.
- **BaseDb abstraction pattern.** Soulforge's memory uses concrete SQLite paths. A clean future direction: define a `MemoryBackend` interface that accepts storage implementations. Agno proves this works cleanly — the agent code never imports SQLite; it imports the abstraction. Not urgent (soulforge is local-first), but the pattern is worth noting in `memory/README.md`.

## What soulforge should NOT copy

- **AgentOS / control plane.** Agno's production runtime is impressive engineering but it's a platform product, not a substrate. Soulforge is a "build your own stack" substrate — adding a bundled runtime would undercut the core positioning. The right move is the opposite: document how to wire soulforge agent code *into* a production service, not ship the service.
- **~100 pre-built integrations (DuckDuckGo, YFinance, etc.).** Agno's breadth is great for demos and fast prototyping. It's wrong for soulforge: CLAUDE.md says "three concrete examples before extracting a shared utility." Bundling integrations in soulforge would create maintenance debt without clear payoff — developers using soulforge are expected to build their own tools.
- **Natural-language team formation.** Agno lets you form teams dynamically at runtime from natural-language descriptions. This is non-auditable by soulforge standards — you can't eval something that is configured at runtime via strings. Soulforge's soul-as-contract approach requires that the team structure is declared up-front and reviewable.
- **Multi-modal as default.** The multi-modal-native design is right for Agno's audience (broad general-purpose agents). For soulforge's audience (coding agents building specific agents), it adds complexity without return. Soul schemas should stay text-first with explicit modality opt-in.
