# Haystack (deepset) — Component Protocol and Pipeline Composition

**Target:** [Haystack by deepset](https://haystack.deepset.ai/) — open-source AI orchestration framework. Canonical sources: [docs](https://docs.haystack.deepset.ai/docs/intro), [GitHub](https://github.com/deepset-ai/haystack).

## What it is

Haystack is a Python framework for building production LLM pipelines and agents. It models everything as a **directed multigraph of typed components**: components declare their inputs and outputs as named, typed sockets, and the pipeline validates connections at build time before any execution begins. Agents are just components with a loop.

## Architecture

- **`@component` decorator protocol.** Any class decorated with `@component` becomes a pipeline node. It must expose a `run()` method that accepts typed keyword arguments (inputs) and returns a `dict` (outputs). Input and output types are declared via `@component.output_types` or `set_output_types()`. The pipeline validates socket compatibility at connection time — a `str` output cannot be wired to an `int` input; this error surfaces before the pipeline runs.

- **Directed multigraph, not a chain.** Pipelines are graphs, not sequential lists. Components can fan out to parallel branches, fan in from multiple sources, and loop back via explicit cycle declarations. The async execution model (`AsyncPipeline`) runs independent branches concurrently without any extra code.

- **`SuperComponent` pattern.** A full pipeline can be wrapped as a single component via `@super_component`. This makes composition explicit: a retrieval-augmented generation pipeline becomes a `RAGComponent` that another pipeline can use as one node. The wrapped pipeline retains all content tracing and debugging signals from its interior.

- **`state_schema` for agent runs.** The `Agent` component accepts a `state_schema`: a typed dict declaration mapping key names to types. Tools in an agent's tool list declare `inputs_from_state` and `outputs_to_state` — they read from and write to named typed keys. State is the explicit contract between the agent loop and its tools, not a bag of text.

- **Tool type hierarchy.** Haystack distinguishes `Tool` (a plain function), `ComponentTool` (a component wrapped as callable), `PipelineTool` (a full pipeline as tool), `MCPTool` (MCP protocol), `Toolset`, `MCPToolset`, `SearchableToolset`. The hierarchy makes the nesting explicit and testable.

- **Pipeline serialization.** Pipelines are YAML-serializable. Components must implement `from_dict`/`to_dict`. This enables no-code pipeline editing and pipeline registries — teams can version and share pipelines as data, not code.

## What soulforge can learn

- **Typed state schema at the soul level.** Soulforge souls currently list `state_keys_written` as untyped name arrays (e.g., in `workflow-orchestrator-soul.md`). Haystack's `state_schema` shows the value of declaring *types* alongside key names. A soul that declares `state_schema: { research_findings: string[], draft: string, fact_check_report: object }` gives a coding agent or endpoint harness a machine-readable contract to validate against before execution. This extends the existing `state_keys_written` convention without replacing it.

- **Build-time socket validation as a design principle.** Soulforge tools validate at runtime. Haystack's insight is that most type mismatches are detectable at *wiring time* (before execution): if tool A returns `{ items: string[] }` and tool B expects `{ item: string }`, that mismatch can be caught when the workflow is assembled, not when it fails at 3am. The `.ai/` guidance layer could encode this: "declare tool output types; the assembling agent validates connections before invoking."

- **SuperComponent pattern → endpoint-as-tool.** The pattern of wrapping a full pipeline as a callable component translates directly: a soulforge endpoint (which may itself chain multiple tools) can be declared as a tool another agent calls. The outer agent sees one typed input/output contract; the inner endpoint's complexity is hidden. This is structurally different from `workflow-orchestrator` (which coordinates specialists via dispatch) — it is a *composability* primitive, not a coordination primitive.

- **AsyncPipeline insight.** Independent branches in a workflow are parallelizable without explicit threading code. Soulforge's `parallel` delegation mode in `workflow-orchestrator-soul.md` is the conceptual equivalent, but the Haystack insight is to make this *default behavior* for independent components rather than an explicit mode choice.

## What soulforge should NOT copy

- **Python decorator magic.** The `@component` decorator relies on Python metaclass inspection to wire up sockets automatically. The TypeScript equivalent would be decorators + reflection, which adds complexity without proportional gain. Soulforge's explicit tool schema (JSON Schema inputs/outputs in the tool module) is already a typed contract — it should be documented as such rather than adding a new decorator layer.

- **YAML pipeline definitions.** Serializing pipelines as YAML adds a second representation that can drift from code. Soulforge's soul markdown is already a more human-readable serialization. The right lesson is not to copy the format but to borrow the discipline: the soul should be the single source of truth for what runs, in the same way a YAML pipeline definition is the single source of truth for a Haystack pipeline.

- **Enterprise RAG weight.** Haystack grew from document indexing (NLP pipelines, Elasticsearch document stores, BM25 retrievers). The framework carries that DNA: `DocumentStore`, `DocumentConverter`, `DocumentSplitter` are first-class citizens. These are irrelevant for soulforge's agent primitives. The component protocol and state schema ideas are extractable; the document infrastructure is not.

- **`from_dict`/`to_dict` requirement on all components.** Requiring every tool to implement serialization methods is friction. Soulforge tools are already versioned TypeScript modules with schema declarations — provenance is captured via observability events, not component serialization.
