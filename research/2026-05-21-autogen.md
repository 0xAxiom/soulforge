# AutoGen 0.4 — Event-Driven Actor Model for Multi-Agent Systems

**Target:** Microsoft AutoGen v0.4+ (current PyPI: 0.7.x line)
- Docs: https://microsoft.github.io/autogen/stable/
- Blog: https://devblogs.microsoft.com/autogen/autogen-reimagined-launching-autogen-0-4/
- GitHub: https://github.com/microsoft/autogen

## What it is

AutoGen is Microsoft Research's multi-agent framework, redesigned in v0.4 around an asynchronous, event-driven actor model. Each agent is an addressable entity that encapsulates its own state and responds only to typed messages delivered by a shared runtime. The framework ships two API layers: `autogen-core` (the actor runtime) and `AgentChat` (higher-level team patterns built on top of it).

## Architecture

- **Actor model at the core.** Agents are micro-actors: they process one message at a time, hold local state, and communicate only through the runtime. No shared mutable state between agents.
- **Typed message handlers via `@message_handler`.** Agents inherit from `RoutedAgent` and declare which message types they process using Python type hints. The runtime dispatches to the right handler — agents don't inspect message type themselves.
- **Two delivery modes: direct send and pub-sub.** Direct send targets a specific `AgentId`. Pub-sub broadcasts a typed event to a named topic; any agent subscribed to that topic receives it. Producers and consumers don't need to know each other.
- **Runtime manages lifecycle, not the developer.** Agents are registered as factories (`MyAgent.register(runtime, "type", lambda: MyAgent())`); the runtime instantiates on first message. Developers don't construct agent instances directly.
- **LayeredAPI:** `autogen-core` → `AgentChat` → `Extensions`. Each layer adds convenience at the cost of flexibility. AgentChat provides canned team patterns (RoundRobin, Selector, Swarm, GraphFlow) built on top of the raw actor runtime.
- **OpenTelemetry built into the runtime layer.** Traces are emitted at the message-passing level, not added as afterthoughts by individual agents.
- **Distributed by default (or at least, designed not to prevent it).** The `SingleThreadedAgentRuntime` is for local dev; the architecture is shaped to allow remote runtimes where agents live on different machines or are written in different languages (Python + .NET parity is an explicit goal).
- **Team patterns are named.** GraphFlow (directed graph of agents), Swarm (tool-driven self-selection), Selector (LLM picks next speaker), RoundRobin. These are first-class named choices, not configuration options.

## What soulforge can learn

- **Typed event topics as a multi-agent coordination primitive.** The existing `handoff-router-soul.md` is point-to-point: router knows all specialists and has one dedicated tool per specialist. The AutoGen pub-sub alternative decouples this — the dispatcher publishes a typed event, and handlers self-register. The soul doesn't need to know its downstream agents at authoring time. This is the right model when the agent population is dynamic or extensible.
  → Add `event-dispatcher-soul.md` to `souls/examples/` demonstrating this pattern for soulforge (done in this commit).

- **Typed message handlers map cleanly to soulforge tools.** AutoGen's `@message_handler(MyMessageType)` is the runtime analogue of soulforge's typed tool input/output schema. Both enforce a contract at the boundary. The lesson: a soul that receives multiple *types* of incoming events should declare them explicitly in its `scope` and match each to a distinct tool — not handle all inputs in one undifferentiated system prompt.

- **Name your team pattern before you write code.** AutoGen forces the choice upfront: RoundRobin? Selector? Swarm? GraphFlow? Soulforge's `ARCHITECTURE.md` already has the "agent loop vs deterministic workflow" decision table. A companion table for multi-agent *coordination* patterns (broadcast-fan-out, point-to-point-handoff, selector-LLM, graph-sequential) would make the same forcing function available to soulforge users at design time.

- **The runtime is infrastructure, not the agent.** AutoGen separates message delivery (runtime) from message handling (agent). Soulforge already separates the soul (policy) from tools (capabilities) and observability (side effects). The parallel: treat the endpoint as the runtime boundary — not part of the soul's identity, just the delivery mechanism.

## What soulforge should NOT copy

- **The layered API proliferation.** `autogen-core` vs `AgentChat` vs `Extensions` creates a three-layer abstraction stack where most users need to understand at least two of the three to debug anything. Soulforge's explicit primitives (souls, tools, memory, eval) stay flat. Don't add a "SoulChat" convenience layer on top — the primitives *are* the API.

- **Python decorator-based routing.** `@message_handler` is elegant in Python but ties the routing contract to the implementation language. Soulforge is language-agnostic — the soul markdown is the contract. Keep routing logic in the soul's `scope`/`refuses` fields and tool schemas, not in decorators that only exist in one runtime.

- **The team-pattern zoo.** Four named team types (RoundRobin, Selector, Swarm, GraphFlow) is already four decisions to make before writing an agent. More patterns will ship. Soulforge should document two durable coordination modes (broadcast/pub-sub and point-to-point/handoff) and let developers compose from there rather than picking from a growing menu of named presets.

- **Memory and planning delegated to the user.** AutoGen's gap (per community analysis) is that it ships no default memory strategy and no default planning layer — developers must wire these themselves. Soulforge already has `memory/` and `tool-planner-soul.md` as primitives. Don't follow AutoGen's lead of leaving these as optional extensions.
