## What it is

Mastra is an open-source TypeScript framework for building production agents, multi-step workflows, and RAG pipelines. It launched in October 2024 from the team behind Gatsby and grew to 300k+ weekly npm downloads by its 1.0 release. Unlike most Python-first frameworks, it is purpose-built for TypeScript and delegates LLM interactions to the Vercel AI SDK rather than implementing its own model layer.

## Architecture

**Two explicit execution models, not one:**

- **Agent** — model-driven loop. The agent reasons about goals, decides which tools to invoke, how many iterations to run, and when to stop. The developer specifies objectives; the agent controls the loop. Exposes `.generate()` (complete before return) and `.stream()` (token-progressive with async resolution of `toolCalls` / `toolResults`).
- **Workflow** — deterministic step graph. The developer wires steps with `.then()`, `.branch()`, `.parallel()`. Each step is created with `createStep()` and carries a Zod `inputSchema` + `outputSchema` — typed contracts enforced at runtime between steps. The developer controls the loop.

The split is the central design decision: agents handle open-ended tasks where the model should decide the sequence; workflows handle tasks where the sequence is known in advance and correctness depends on determinism.

**Memory in three tiers:**

- **Conversation history** — raw user/assistant/tool-result turns. Grows unboundedly; memory processors can trim or filter before it hits the context window.
- **Working memory** — persistent structured facts (user name, preferences, goals). Survives across conversation threads. Updated by the agent, not by raw message ingestion.
- **Semantic recall** — vector-backed retrieval of past interactions. Finds contextually relevant prior exchanges by meaning, not keyword.

Memory is scoped at two levels: **resource** (stable cross-thread identity — the user or entity) and **thread** (single conversation). Sub-agents spawned by a supervisor get a fresh `threadId` but inherit the stable `resourceId`, so they start clean but retain working memory.

**Eval as scorers attached to agents and steps:**

Scorers are functions returning a float in [0, 1]. They can be model-graded (LLM-as-judge), rule-based (deterministic logic), or statistical. Scorers attach directly to an agent or to individual workflow steps, run asynchronously without blocking execution, and write scores to a persistent database for CI and longitudinal tracking. Sampling rate is configurable (0–100%).

**Human-in-the-loop via suspend/resume:**

Workflows can suspend mid-execution, persist state, and resume from the last active step. The suspended state is explicit in the result object (`status: "suspended"`), not a magic exception. This enables long-horizon workflows that require human approval before continuing.

**MCP integration is first-class:**

Agents can expose themselves as MCP servers and consume tool sets from MCP providers. This makes Mastra agents composable with other MCP-aware systems without writing adapter code.

## What soulforge can learn

**1. The agent/workflow split deserves explicit documentation in ARCHITECTURE.md.**
Soulforge souls don't distinguish between "I control the sequence" and "the model controls the sequence." The tool-planner soul is about planning *before* tool use, but says nothing about when a deterministic step graph is a better primitive. The Mastra model is clean: if the sequence is known, encode it as steps with typed I/O; if it isn't, use an agent loop. Soulforge's ARCHITECTURE should state this explicitly as a design decision.
→ Applies to: `docs/ARCHITECTURE.md`, `souls/examples/`

**2. Typed step handoff records are a missing souls pattern.**
When one agent hands state to another (planner → executor, researcher → writer), the handoff payload has no typed contract in soulforge today. Mastra's `createStep` with Zod `inputSchema` / `outputSchema` per step shows the value: if each step declares its expected input and output shape, the chain is checkable before running it. A soulforge "workflow soul" should define the handoff record shape explicitly in its `memory` section.
→ Applies to: `souls/schema/soul.schema.json` (handoff_record field), `souls/examples/`

**3. Resource-scoped vs. thread-scoped memory is a useful split.**
Soulforge's memory README documents long-term vs. short-term memory, but doesn't name the resource/thread scoping pattern. Sub-agents that share a user identity but need clean conversation histories benefit from this split. The pattern is: stable `resourceId` carries working memory; fresh `threadId` isolates conversation history per task.
→ Applies to: `memory/README.md`

**4. Suspend/resume for long-horizon souls.**
Soulforge's long-horizon monitor pattern (from the task routing table) doesn't say how to handle mid-execution pauses. Mastra's suspend/resume model is simple: produce a checkpoint record, store it, resume from there. For soulforge, this maps to a memory checkpoint + an explicit status field in the soul's output contract.
→ Applies to: `memory/`, `souls/` long-horizon pattern

**5. Async scorers don't block the hot path.**
Soulforge's eval README leaves open how to run eval without slowing agents down. Mastra's answer: scorers run asynchronously, after the response is delivered, sampling only a fraction of calls. Soulforge eval could adopt the same: attach scorers to observability events, run out-of-band, write to the JSONL trace sink.
→ Applies to: `eval/`, `observability/`

## What soulforge should NOT copy

**The framework runtime.** Mastra is an opinionated runtime — `new Mastra({ agents, tools })` wires everything together at startup. Soulforge is explicitly not a runtime. The `createTool` / `createStep` / `createWorkflow` factory pattern only makes sense inside a Mastra instance. Porting these factories would create the framework CLAUDE.md forbids.

**Provider routing and model abstraction.** Mastra connects to 40+ providers through a unified interface (delegating to AI SDK). Soulforge is deliberately provider-agnostic at the schema level and doesn't want to own that routing. Let AI SDK or direct provider calls handle it in examples.

**The Mastra Studio UI.** Mastra ships a web studio for trace inspection and experiment management. Soulforge's observability is file-based JSONL — appropriate for a copy-pasteable substrate, not a hosted product. A studio would require a daemon, auth, and persistence infrastructure that violates local-first operability.

**Opinionated directory structure.** Mastra enforces a file layout. Soulforge's bet is that the AI coding agent navigates structure via `.ai/repo-map.json`, not by convention enforcement. Don't add required directory layouts.
