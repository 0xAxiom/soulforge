# ElizaOS

**Target** — [elizaos.github.io](https://github.com/elizaOS/eliza) · [docs.elizaos.ai](https://docs.elizaos.ai) · [arxiv paper](https://arxiv.org/html/2501.06781v1) · v2.0.3, May 2026

## What it is

ElizaOS is a TypeScript framework for building AI agents that operate primarily on social platforms and Web3 networks. The canonical use case is a persistent agent that lives on Discord, Telegram, Farcaster, and Twitter simultaneously, maintains a distinct personality, holds long-term memory of users and conversations, and can execute blockchain transactions. With 18.5k GitHub stars and 297 releases, it is the dominant framework for socially-deployed crypto agents.

The guiding philosophy is explicit: "Roughness is better." Ship features, accept imperfection, grow the ecosystem. This is the explicit opposite of soulforge's substrate philosophy.

## Architecture

- **Character file as JSON identity contract.** An agent's entire definition lives in a single JSON file: bio, lore, example messages, style guidelines, model provider selection, plugin list, and `secrets` (API keys, private keys). The character file is the agent — not a config for a framework's runtime. Swapping character files swaps agents. Private keys are character-scoped, not env-var-global; this bounds credential blast radius to the agent identity.

- **AgentRuntime as the coordination hub.** A single `AgentRuntime` class handles message ingestion, state assembly, memory retrieval, action selection, LLM call, and response dispatch. The runtime is not exposed to the agent — it wraps the agent. State is composed fresh on each turn from: current message + retrieved memories + provider-injected context + conversation history.

- **Action / Provider / Evaluator / Service — four roles, not two.** Most frameworks name tools (DO) and prompt (SEE) but ElizaOS makes all four explicit:
  - **Actions** — capabilities the agent can execute (token swap, image post, API call)
  - **Providers** — context suppliers that run *before* the LLM call to inject real-time data (wallet balance, market price, user relationship data). Providers are read-only; they don't execute side effects.
  - **Evaluators** — post-response analyzers that extract facts, update long-term memory, and track goal progress. They run *after* each response, in production, not in a test harness.
  - **Services** — long-lived connections to external systems (WebSocket connections, polling loops, Discord gateway). Services outlive individual turns.

- **Plugin interface is flat and declarative.** `Plugin = { name, description, actions[], providers[], evaluators[], services[], init(), dbAdapters, routes, events }`. Registration order is deterministic; priority field controls load sequence; dependency declarations ensure prerequisite plugins load first. A plugin can contribute to any or all four roles.

- **Memory in four layers.** Messages (conversation history per room/user), Knowledge (vectorized documents, RAG retrieval), Facts (extracted per-entity attributes — "user prefers ETH over SOL"), Lore (static world-context embedded in the character file itself). These map to: working memory, retrieval, long-term learning, and background context. Evaluators populate Facts from conversation. Knowledge uses contextual embeddings (Anthropic's contextual retrieval technique).

- **Clients = social platform adapters.** A "Client" is an inbound/outbound channel connector: Discord, Telegram, Twitter/X, Farcaster, Slack. Each client translates platform-specific message objects into the runtime's internal `Message` type and routes responses back through platform APIs. The agent code is platform-agnostic; the client is the protocol adapter.

- **90+ plugins, community-maintained.** DeFi connectors, NFT tools, TTS, image gen, sentiment analysis, blockchain data providers. Breadth over depth.

- **Stateful-by-design (opposite of Agno).** Agno's 2μs-per-agent perf comes from statelessness. ElizaOS's runtime holds connection state, open WebSocket clients, and memory cache. This is the correct choice for persistent social agents that need fast response times — rebuilding those connections per-message would be prohibitive. The tradeoff: harder to scale horizontally.

## What soulforge can learn

- **Name "Provider" as a distinct primitive.** Soulforge has `tools/` (DO) and `memory/` (state). It doesn't name the pattern of "read-only context supplier that runs before the main model call." A Provider is not a tool (no side effects, no user-facing capability) and not memory (not stored, just computed). It's real-time context injection: current time, user relationship data, live prices, feature flags. Soulforge should name this in `docs/ARCHITECTURE.md` — even if the implementation is just a function call in the tool harness, the *concept* separates "what the agent knows from state" from "what the agent knows from the world right now."

- **Name "Evaluator" as distinct from offline eval.** Soulforge's `eval/` module is for offline testing of agent behavior. ElizaOS's evaluators are *in-production continuous memory updaters* — they run after every response to extract facts and update long-term memory. These are not the same concept. Soulforge's `tiered-memory-soul.md` and `reflect.ts` do a version of this, but the pattern isn't named. Calling it "production evaluator" (vs "test evaluator") clarifies when a developer should reach for each.

- **Social channel ingress has a different shape than HTTP.** Soulforge's endpoint examples are all HTTP. Social message ingress is different: the "request" carries sender identity + channel/thread ID + platform + message history slice, not a JSON body. A soul handling a Telegram or Farcaster message needs to declare its context shape explicitly (who sent it, what channel, what's the recent history window, what are the platform reply constraints). Soulforge should have an example that demonstrates this shape.

- **Character secrets scope credentials to identity.** ElizaOS puts `secrets` inside the character file. This means each agent identity has its own bounded credential set, not a shared env pool. When you spawn multiple agents, they can't accidentally cross-use credentials. Soulforge doesn't address this pattern — it defers to "use env vars" without noting the identity-scoping concern.

- **Services as a named primitive for long-lived connections.** Soulforge's tools are assumed to be stateless (called, return, done). Some agent capabilities require persistent connections: WebSocket listeners, polling loops, subscription streams. Naming these as "Services" — separate from tools — clarifies which capabilities are per-call and which are lifecycle-bound.

## What soulforge should NOT copy

- **JSON character files over markdown souls.** ElizaOS's character is JSON because it's machine-ingested by the runtime. Soulforge's souls are markdown because they're *read by developers and coding agents for policy*. CLAUDE.md is explicit: "Souls are markdown, not YAML." The tradeoff is real — JSON validates, markdown communicates intent. For a substrate whose primary user is a developer or coding agent reading a policy document, markdown wins.

- **"Roughness is better" philosophy.** ElizaOS ships features at community pace and accepts rough edges. Soulforge's CLAUDE.md requires end-to-end working examples and no half-finished modules. These are irreconcilable — soulforge's substrate promise is that examples run, so developers can copy-paste rather than debug.

- **90-plugin breadth.** The plugin ecosystem is great for demos and for a community that wants batteries-included. For soulforge's audience (developers building specific agents), broad plugin libraries create maintenance debt and trust problems (who audits plugin X?). The three-concrete-examples rule before abstraction is the right counterweight.

- **Stateful-by-design runtime.** ElizaOS holds WebSocket connections and memory caches across turns because it's running persistent social agents. Soulforge is a substrate — it doesn't run agents. Recommending stateful runtime design would conflict with the "no startup state" invariant surfaced in the Agno research. For ephemeral or serverless deployments, stateless construction is correct. Soulforge should document both patterns, not default to either.

- **Web3-first action inventory.** Blockchain transactions as default actions shapes the entire plugin ecosystem. Soulforge is general-purpose; economic actions are explicit opt-in via the economic boundary documented in ARCHITECTURE.md.
