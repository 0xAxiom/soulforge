# Semantic Kernel → Microsoft Agent Framework

**Target:** [Microsoft Agent Framework (MAF)](https://devblogs.microsoft.com/foundry/introducing-microsoft-agent-framework-the-open-source-engine-for-agentic-ai-apps/) — formerly Semantic Kernel, converged with AutoGen in Oct 2025, GA in Q1/Q2 2026.
Repo: [github.com/microsoft/semantic-kernel](https://github.com/microsoft/semantic-kernel)
Docs: [learn.microsoft.com/semantic-kernel](https://learn.microsoft.com/en-us/semantic-kernel/overview/)

## What it is

An enterprise-grade multi-agent orchestration SDK (C#, Python, Java) that merges Semantic Kernel's plugin/kernel/middleware model with AutoGen's research-grade multi-agent patterns. It is a runtime you import and instantiate — the opposite of soulforge's substrate approach. Backed by Microsoft, used in production by Fortune 500 companies on Azure Foundry.

## Architecture

- **Kernel** — dependency injection container that holds all services (LLM connectors, logging, HTTP) and plugins (callable functions). Every prompt or tool call routes through it. This is the central bus.
- **Plugins** — callable units exposed as: native code functions, prompt templates, OpenAPI specs, or MCP servers. The kernel can itself be exported as an MCP server. This collapses the tool/function/API distinction into one vocabulary.
- **Filters / Hooks** — middleware that intercepts at specific points in the execution pipeline: `before_function_call`, `after_function_call`, `before_prompt_render`, `after_prompt_render`, `before_agent_turn`, `after_agent_turn`. These are the enforced observability, safety, and logging injection points — not optional add-ons.
- **Orchestration patterns** — unified API across five patterns: Sequential (pipeline), Concurrent (fan-out), Handoff (dynamic routing by context), GroupChat (collaborative brainstorm), and MagenticOne (complex multi-agent generalist). You swap patterns without rewriting agent logic.
- **Two orchestration modes** — Agent Orchestration (LLM drives decisions, creative/adaptive) vs Workflow Orchestration (deterministic, business-logic driven). Explicit duality, not a single abstraction that tries to cover both.
- **CodeAct** — collapses multi-step tool calls into a single executable code block generated by the LLM. ~50% latency reduction, ~60% token reduction. The LLM writes a program rather than driving a loop.
- **FIDES** — information flow control as middleware. Assigns integrity and confidentiality labels to content flowing through the kernel. Prevents prompt injection by tracking which content is trusted vs untrusted throughout execution.
- **A2A Protocol** — cross-agent interoperability. Agents publish A2A endpoints; other agents call them without coupling to implementation.
- **Foundry Hosted Agents** — cloud deployment with managed session state, identity, autoscaling, versioning, and built-in observability. The runtime can be local (InProcessRuntime) or managed (Foundry).

## What soulforge can learn

**Filters as first-class intercept contract** — SK made explicit the intercept points every agent system actually needs. Soulforge's observability is sink-based (emit events after the fact). It has no formalized notion of *before* intercepts. A soul that declares `pre_execution_checks` and `post_execution_checks` as a first-class field — not just narrative policy — would let coding agents know where to wire obs hooks and safety checks consistently. Add this to `souls/schema/soul.schema.json` as optional structured fields.

**The filter → eval pipeline connection** — SK's `after_function_call` filter is the right moment to emit to both observability (latency/cost) and eval (capture actual vs expected output). Soulforge's observability and eval are currently separate concerns with no declared bridge. The filter pattern shows where the bridge lives: at the tool-call boundary, not at the turn boundary.

**CodeAct for multi-step tools** — if a soul's `tools` section involves a sequence of dependent calls, a CodeAct variant would reduce round-trips. This isn't about adopting MAF's runtime, it's about recognizing that "generate a small program, execute it once" is a superior pattern to "call tool A, observe, call tool B, observe" for deterministic sequences. Worth a soul example and a note in `tools/README.md`.

**Two-mode orchestration taxonomy** — the LLM-driven vs deterministic split is clean and should be documented in soulforge's ARCHITECTURE.md. Right now soulforge has `deterministic-workflow-soul.md` and `workflow-orchestrator-soul.md` without explicitly naming this axis. Naming it helps coding agents choose the right soul and helps humans understand the tradeoff.

**Plugin = capability regardless of origin** — SK's plugin concept unifies native functions, prompt templates, and OpenAPI endpoints under one noun. Soulforge has `tools/` (native) and `endpoints/` (HTTP) as separate primitives. This is the right decomposition for a substrate, but the README should note that "endpoint as tool" is a valid composition pattern — an agent that calls a soulforge endpoint is just using it as a plugin.

## What soulforge should NOT copy

- **The Kernel DI container** — SK's central kernel works because it's a runtime you import. Soulforge explicitly has no runtime. A "kernel" abstraction would break the substrate model and add a layer coding agents don't need — they already have a file system and CLAUDE.md as their navigation substrate.
- **Multi-language SDKs** — C#, Python, Java support is an enterprise requirement driven by existing corporate codebases. Soulforge is TypeScript-native because TypeScript is the right tool for the coding-agent-first audience. Adding language parity would be complexity without users.
- **Dependency injection architecture** — DI makes sense in a long-running enterprise runtime with injected services per request. Soulforge agents are instantiated from markdown policy, not from a service graph. DI would make the soul-to-code path more opaque, not less.
- **Enterprise lifecycle overhead** — versioned kernel objects, scoped injection, disposal patterns, health checks per service — all appropriate for Azure-deployed microservices, wrong for agents built from a substrate repo. Soulforge's answer to lifecycle is "commit + deploy"; it should stay that way.
- **GroupChat orchestration as default** — GroupChat (all agents participate in one conversation) looks attractive but produces verbose, hard-to-debug multi-agent systems. The Handoff pattern is almost always preferable when domain boundaries are clear. Don't normalize GroupChat as a starting point; document it as a last resort.
